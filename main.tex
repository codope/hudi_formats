\begin{abstract}
As lakehouse tables scale toward millions of files and partitions, metadata itself becomes big data.
Traditional big data systems try to reduce metadata to keep management tractable, compromising query performance; cloud data warehouses like Snowflake, BigQuery, and Redshift instead treat metadata as a managed, indexed structure---but within proprietary architectures.
Apache Hudi brings this principle to the open lakehouse, treating \emph{metadata as data}: an internal Merge-on-Read table whose partitions implement a multi-modal indexing subsystem---column statistics, bloom filters, record-level, secondary, and expression indexes---all transactionally coupled with data commits and stored in an SSTable-like layout (HFile) optimized for the point and prefix lookups that dominate query planning.
Concurrent indexing via a three-phase protocol backfills large indexes online while writers continue to commit.
We summarize the design and report improvements on both synthetic benchmarks (TPC-DS) and Uber production workloads, demonstrating over 99\% reduction in data scanned (secondary index on a 1.8\,TB table), record-level index lookups 64$\times$ faster than global simple index at 36\,billion-row scale with sub-millisecond per-key latency even at 400\,billion+ rows, and 72\% lower write latency on upsert workloads.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Open table formats have become the transactional layer for lakehouse deployments on cloud object storage \cite{zaharia2021lakehouse}.
At large scale, metadata becomes the bottleneck: directory listings and bulk footer/stat reads dominate planning latency and API cost \cite{armbrust2020deltalake}.
Production deployments underscore the magnitude: Uber manages 19,500+ datasets (350\,PB, 6\,trillion rows/day) on Apache Hudi \cite{uber-hudi-blog}, and ByteDance operates an exabyte-scale lake \cite{bytedance-hudi-blog}.
As systems reach this scale, metadata requires database-style techniques for storage, indexing, and maintenance \cite{edara2021bigmetadata}.

In-file statistics such as Parquet column-chunk min/max values are necessary but insufficient: the total footer metadata is $O(\#\textit{files} \times \#\textit{columns})$, and every footer must be opened to perform file selection \cite{zeng2024columnar}.
For wide tables with thousands of columns, this forces planners to parse far more metadata than any single query requires.
External index structures that decouple file selection from footer reads are essential \cite{liu2023openformats}.

Apache Hudi addresses this by treating metadata as first-class data.
Cloud data warehouses have long maintained metadata as a managed, indexed structure---Snowflake with zone maps in a dedicated service \cite{dageville2016snowflake}, BigQuery with distributed statistics tables \cite{edara2021bigmetadata}, Redshift with in-memory zone maps \cite{armenatzoglou2022redshift}---but these rely on proprietary, always-on metadata services tightly coupled to a single engine.
An open lakehouse has no such service: metadata must reside alongside data on object storage, remain readable by any engine, and stay consistent without a centralized coordinator.
Hudi brings this principle to the open lakehouse: each table maintains an internal \emph{metadata table}, co-located with data files, that stores file listings and multiple index modalities---replacing recursive listings and broad footer scans with index-based planning \cite{hudi-metadata,onehouse-mmi}.
Hudi further supports \emph{concurrent indexing} to build large indexes online without pausing ingestion \cite{hudi-indexing}.

\textbf{Contributions.}
We present a metadata-as-data architecture for open table formats: an internal MoR table whose partitions form a multi-modal index substrate with an HFile-based layout optimized for point and prefix lookups, and a concurrent indexing protocol that builds indexes online without blocking writers.
We evaluate on TPC-DS and Uber production tables at up to 400\,billion+ rows.

\paragraph{Organization.}
The metadata table is a Merge-on-Read (MoR) Hudi table where each metadata or index type is a distinct partition (see Table~\ref{tab:modalities}) \cite{hudi-metadata,hudi-indexes}.
MoR is the natural fit because every data commit generates metadata updates; rather than rewriting base files on each commit, MoR appends delta logs and compacts in the background, ensuring that ingestion latency is not penalized by metadata maintenance \cite{hudi-metadata}.

\paragraph{Transactional consistency.}
Hudi translates each data table write into metadata table updates and couples the two timelines so they remain in sync (Figure~\ref{fig:mdt-partitions-keys}) \cite{hudi-metadata}.
A write is visible only when \emph{both} the data table and metadata table are committed, preventing partial exposure of data or index state under failures \cite{onehouse-mmi}.

\paragraph{HFile for point and prefix lookups.}
Lakehouse metadata access is dominated by \emph{point} and \emph{prefix} lookups (e.g., ``stats for column \texttt{c} in file \texttt{f}'', ``all blooms for partition \texttt{p}'').
Hudi stores metadata partitions in an SSTable-like format (HFile), maintaining records in sorted order with a block index for point reads and range scans \cite{hudi-techspec}.
Published experiments report 10--100$\times$ lower point-lookup latency for HFile compared to Parquet/Avro at 10M-entry scale \cite{onehouse-mmi}.
At Uber, the metadata table is deployed across 90\% of production datasets; for the largest tables exceeding 300 billion rows, the record index alone is sharded into up to 10,000 HFiles \cite{uber-hudi-blog}.

\paragraph{Prefix key design for wide tables.}
For column statistics, Hudi composes record keys by concatenating \emph{column name}, \emph{partition name}, and \emph{data file name}, enabling prefix scans.
Planners issue prefix lookups only for columns referenced by a query, reducing work from $O(\#\textit{table columns})$ to $O(\#\textit{query columns})$ \cite{onehouse-mmi}.

\section{Multi-Modal Indexing}
\label{sec:modalities}

Hudi's metadata table acts as an \emph{index substrate}: index modalities are materialized as partitions and can be enabled/disabled independently as workload needs evolve \cite{hudi-metadata,hudi-indexes}.
Table~\ref{tab:modalities} summarizes key modalities and their database analogues.

\toprule
\textbf{MDT partition / modality} & \textbf{DB analogue} & \textbf{Primary effect} \\
\midrule
\texttt{record\_index} (global RLI) & primary index & fast upserts/deletes; point lookups \\

\bottomrule

\paragraph{Record-level index.}
The record-level index (RLI) maps record keys to data-table locations in the \texttt{record\_index} partition \cite{hudi-indexes,hudi-techspec}.
This is the lakehouse analogue of a \emph{primary index}: it enables efficient upserts and deletes by avoiding full file-group scans to locate candidate records.
To accommodate large key spaces, the record index uses hash-based sharding and supports point lookups for keyed reads \cite{hudi-indexes,hudi-rli-blog}.

\paragraph{Column statistics and data skipping.}
The \texttt{column\_stats} partition stores per-file, per-column aggregates (min, max, null count, value count) in the metadata table.
Query planners evaluate predicates against these statistics to skip files whose value ranges provably exclude matching records---the same principle as zone maps in cloud data warehouses \cite{dageville2016snowflake,edara2021bigmetadata,armenatzoglou2022redshift}.
This coarse-grained pruning composes with bloom filters and secondary indexes to progressively narrow the candidate file set.

\paragraph{Expression index.}
An expression index indexes a function of a column, stored as an \texttt{expr\_index} partition \cite{hudi-indexes,hudi-techspec}.
This is the lakehouse analogue of indexes on expressions in PostgreSQL \cite{pg-expr-index}: it creates an access path for predicates that do not match raw columns.

\paragraph{Partition evolution.}
A table partitioned by day cannot prune at finer granularity without repartitioning; an expression index on an hour-extraction function provides hour-level pruning without changing the physical layout.

\paragraph{Adaptive bloom filters.}
Hudi supports \emph{adaptive bloom filters} for membership testing during upserts, adjusting filter size per file to maintain a configured false-positive ratio \cite{hudi-indexes}.
This matters for streaming and skewed workloads where file-group cardinalities vary and static filters would saturate.

\paragraph{Secondary index.}
The secondary index accelerates predicates on non-key columns via a two-hop lookup---secondary value $\rightarrow$ record key $\rightarrow$ file location---reusing RLI rather than maintaining a separate file-location mapping \cite{hudi-techspec}.

\section{Concurrent Indexing}
\label{sec:concurrent}

Indexes increase write cost because they must be maintained as data changes.
In a continuously ingested lakehouse table, pausing writers to build a new index would violate ingestion SLAs.
Traditional databases perform online index builds under row-level locks, but lakehouse writers operate over immutable file groups on object storage, where coordination must happen at the commit-metadata level.
Hudi addresses this with asynchronous \emph{concurrent indexing} that builds indexes while writers continue to commit (Figure~\ref{fig:concurrent}) \cite{hudi-indexing}.

  \label{fig:concurrent}
  \Description[Concurrent indexing timeline]{Swimlane timeline with two lanes: Writer lane showing four commits (c1 through c4) on a horizontal arrow, and Indexer lane below showing three phases: Schedule (dark gray, takes brief lock at commit c1), Execute (light gray, no lock, runs while c2 and c3 are committed), and Reconcile (dark gray, validates c2 and c3 under lock). A vertical dashed line marks the indexing instant.}
\end{figure}

Concurrent indexing uses a three-phase protocol that combines optimistic and multi-version concurrency control techniques:
(1) \textbf{Schedule} snapshots all completed commits up to an \emph{indexing instant} under a brief metadata-table lock, establishing a consistent baseline;
(2) \textbf{Execute} materializes index entries from the snapshotted files without holding any lock---this is the expensive phase (scanning base files, computing index keys), and writers continue committing unblocked; and
(3) \textbf{Reconcile} reacquires the metadata-table lock, identifies commits that landed during execution ($c_2$, $c_3$), and applies catch-up index updates for those commits before finalizing.

\paragraph{Consistency invariant.}
Because metadata updates are transactionally coupled to data commits (\S\ref{sec:mdt}), reconciliation can deterministically replay missed commits: each data commit records exactly which files were added or removed, and the indexer applies the corresponding index mutations.
If a conflict is detected---e.g., a commit was rolled back between schedule and reconcile---the indexing attempt aborts and retries from a fresh snapshot, ensuring no partial index state is ever visible to readers.
After the initial build, subsequent data commits maintain the index incrementally through the same transactional coupling, so the concurrent protocol is needed only for first-time index creation or full rebuilds.

\section{Evaluation}
\label{sec:eval}

\paragraph{Setup.}
The secondary index benchmark uses Hudi with Spark on a 1.8\,TB TPC-DS \texttt{store\_sales} table (286K files, 7.2\,billion rows).
The RLI lookup benchmark uses Spark with 1,000 executors on Uber production tables stored on GCS.

\paragraph{Secondary index.}
A secondary index on the \texttt{store\_sales} table reduces data scanned by over 99\% (759\,GB to 593\,MB) and files read from 286,603 to 150, improving warm-run query latency by 96\% (Table~\ref{tab:tpcds}).

\paragraph{Record-level index.}
Global simple index joins incoming keys against every file group, so its cost grows with table size.
RLI resolves keys via point lookups into sharded HFiles (Table~\ref{tab:rli}): at 36B rows the speedup is 64$\times$, and at 400B+ rows RLI maintains sub-millisecond per-key latency while global simple fails to complete.
On a separate 1\,TB upsert workload, RLI also reduces write latency by 72\% and data shuffled by 92\% \cite{hudi-rli-blog}.

\paragraph{Production evidence.}
Of Uber's 19,500+ Hudi datasets (\S\ref{sec:intro}), 3,600 upsert-heavy tables rely on the record index in production, with per-key lookup latency of 1--2\,ms from sharded HFiles \cite{uber-hudi-blog}.

\paragraph{Limitations.}
Cross-format benchmarks against Delta Lake and Apache Iceberg indexing remain important future work.

\section{Discussion}

All three major open table formats address the metadata bottleneck with different architectural choices.
Delta Lake consolidates file-level statistics in a transaction log with periodic Parquet checkpoints \cite{armbrust2020deltalake}.
Iceberg organizes snapshots as a manifest hierarchy with per-file column statistics in manifests and Puffin sidecar files for bloom filters and sketches \cite{iceberg-spec}.

Hudi extends the metadata-management idea into a generalized index substrate.
Rather than a log or manifest tree, Hudi stores metadata in an internal MoR table where each index modality (record-level, secondary, expression) is an independently managed partition \cite{hudi-metadata,hudi-indexes}.
The HFile-based layout optimizes for point and prefix lookup patterns, and the concurrent indexing protocol (\S\ref{sec:concurrent}) provides an explicit mechanism for building indexes online \cite{hudi-indexing}.
This positions the metadata table as an extensible, lookup-oriented index structure that borrows from database indexing principles.
In enterprise data lakes where records flow through multiple transformation stages with upserts at each stage, such a substrate directly reduces the cost of locating and updating records across the pipeline.

\section{Conclusion}

At exabyte scale, metadata management is a data-management problem.
Hudi's metadata-as-data design stores metadata and indexes as a first-class internal MoR table, couples updates via multi-table transactions, and organizes index modalities as independently maintained partitions.
The metadata table storage format optimizes for the point and prefix lookup patterns that dominate lakehouse query planning, and concurrent indexing enables online backfill without pausing writers.
Future directions include bitmap and vector indexes for broader predicate types, cost-based index selection, and cross-format index interoperability.
As tables grow, metadata must be treated as managed, indexed data, and we believe the metadata-as-data pattern offers a principled foundation for this evolution.
